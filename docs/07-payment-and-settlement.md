# 07. 결제 · 정산

## 11-5. 결제

### 요구사항
- 부모 결제 가능
- 학생 정보와 결제자 정보 분리 가능
- 예약 확정과 결제 순서 정책 정립 필요
- 환불/취소 규정 반영

### MVP 정책
- 부모가 결제하되 실제 멘티 정보는 학생 기준으로 저장 가능해야 함
- 결제 UX는 최대한 단순화

## 11-10. 정산

### 정산 조건
- 상담 완료 처리
- 취소/환불/클레임 여부 반영
- 플랫폼 수수료 차감
- 멘토 정산 금액 확인 가능

### 초기 정책
- 50:50 기준 검증
- 실제 운영 데이터로 조정 가능
- 멘토 가격은 자율 책정 (관리자 설정 최저가~최고가 범위 내), 수수료율은 가격과 무관하게 동일 50% 적용
- 쿠폰 할인은 멘토 수익을 감소시키지 않음 (플랫폼 부담, 정산 시 멘토 기준가 기준으로 산정)

## 21. 결제/영수증/쿠폰 (v1.1)

### 21-1. PG
- **토스페이먼츠 단독** (포트원 미사용)
- 쇼핑몰 상점(MID `ajaschqcno`)과 분리된 **멘토링 전용 MID 발급** (정산/세무 분리 목적)
- 지원 수단: 카드, 간편결제(토스/카카오/네이버페이), 계좌이체
- 개발 환경 분리: 웹훅 엔드포인트 `/api/webhooks/toss-mentoring`, API 키는 MID별 별도 발급 (Spring Boot 프로파일로 관리)
- 정산 계좌: 쇼핑몰과 **별도 계좌 권장** (재무 분리 목적)

### 21-1-1. MID 발급 체크리스트 (계약 검토)

- 카드/간편결제/계좌이체 수수료율이 기존 MID와 동일/상이 여부
- 월 고정비·진입수수료·보증금 유무 (토스페이먼츠는 대체로 없음)
- 정산 주기 (D+2, D+3) 및 정산 수수료
- 현금영수증 기능 활성화 (1만원 이상 용역 의무)
- 웹훅 URL, API 키 MID별 별도 발급 확인
- 토스 관리자 페이지에서 결제UI·약관·회사정보(서비스 URL) 멘토링 전용으로 설정

### 21-2. 현금영수증
- 1만원 이상 용역은 발급 의무
- 결제 시 현금영수증 신청 UI 포함
- 국세청 자동 연동

### 21-3. 영수증/거래내역
- 결제 완료 후 마이페이지에서 영수증·거래내역서 다운로드 가능

### 21-4. 쿠폰/할인 (v1.2 — ADR-002로 §25-7 클로즈)

#### MVP 분류
| 영역 | MVP 분류 | 비고 |
|---|---|---|
| **쿠폰 발급** | **OUT-W5** | 운영자 수동 SQL/엑셀로 직접 INSERT (`coupons`·`coupon_redemptions`). **클레임 보상 한정** |
| **쿠폰 사용 (멘티)** | **IN-수동** | 결제 화면에서 코드 입력 → 검증 API 호출 → 할인 적용 → 결제 금액 차감 |
| **쿠폰 자동 발급** | OUT-W5 | 첫 상담 자동 할인·생일 쿠폰 등 자동 트리거는 W5+ |
| **쿠폰 자동 만료 알림** | OUT-W5 | A09 등 별도 알림 코드 미신설. 운영자가 만료일 직접 SQL 설정 |
| **쿠폰함 (마이페이지)** | IN-수동 | 보유 쿠폰 리스트 + 만료일 + 사용 상태 표시 (mockups/mypage/coupons.html) |

#### 운영 동선
- 발급: 운영자가 클레임 처리 또는 1:1 문의 답변 시 SQL/엑셀로 직접 쿠폰 row 생성 → 코드를 1:1 답변 본문에 첨부 전달
- 멘티 사용: 결제 화면 "쿠폰 코드" 입력란 → 서버 검증(존재·유효기간·미사용) → 할인 금액 표시 → 결제 진행
- 정산 영향: **멘토 정산은 멘토 기준가(상품 가격) 기준 50:50** 그대로 유지. 쿠폰 할인은 **플랫폼 부담** (멘토 수익 감소시키지 않음)

#### 데이터 모델 단순 스키마 (MVP)
```sql
CREATE TABLE coupons (
  code VARCHAR(32) PRIMARY KEY,
  discount_type VARCHAR(10) NOT NULL CHECK (discount_type IN ('FIXED','PERCENT')),
  discount_value INTEGER NOT NULL,
  expires_at TIMESTAMP,
  issued_to_user_id BIGINT REFERENCES users(id),  -- 발급 대상 (단일 사용자 한정 시), NULL = 누구나
  issued_by_admin_id BIGINT REFERENCES users(id),
  issue_reason TEXT,
  created_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE TABLE coupon_redemptions (
  coupon_code VARCHAR(32) NOT NULL REFERENCES coupons(code),
  used_by_user_id BIGINT NOT NULL REFERENCES users(id),
  booking_id BIGINT NOT NULL REFERENCES bookings(id),
  discount_amount INTEGER NOT NULL,
  used_at TIMESTAMP NOT NULL DEFAULT NOW(),
  PRIMARY KEY (coupon_code, used_by_user_id)  -- 중복 사용 방지
);
```

#### 운영 규칙
- **중복 사용 방지**: `coupon_redemptions(coupon_code, used_by_user_id)` 복합 PK로 1인 1회 강제
- **만료 검증**: 결제 화면 검증 시 `expires_at > NOW()` 체크. 만료 자동 알림 미구현
- **부분·전액 할인**: 정액(`FIXED`) / 정률(`PERCENT`) 단순 구조. 최소 결제 금액 ≥ ₩0 (음수 불가)
- **최저가 충돌**: 쿠폰 적용 결과가 `price_min` 미만이어도 결제는 허용 (할인은 플랫폼 부담이라 멘토 수령액에 영향 없음)
- **환불 시 쿠폰 처리**: 멘티 100% 환불 시 쿠폰은 사용 처리 그대로 유지(재사용 불가). 멘토 유책 100% 환불 시도 동일. 운영자가 클레임 보상으로 새 쿠폰 발급 가능

#### 라이프사이클 라벨

마이페이지 쿠폰함·관리자 화면의 표시값(`UNUSED`·`USED`·`EXPIRED`)은 [docs/14 §5-6 `coupon_status`](./14-enum-catalog.md#5-6-coupon_status-couponsstatus--adr-002) 카탈로그를 SoT로 사용. 물리 컬럼은 미도입(redemption 적재 + `expires_at` 비교로 도출), API 응답·UI 단계에서만 라벨 매핑.

### 21-5. 정산 주기 (확정)
- **주간 정산**: 매주 월요일 전주(월~일) 완료 건 정산
- 정산 계산 기준일: 상담 완료 확정 + 환불 가능 기간(3일) 경과일
- 정산 상세 화면에서 건별 내역/수수료/세금 확인 가능
- 은행 영업일 고려: 정산일이 공휴일이면 다음 영업일

### 21-6. 결제 실패 · 재결제 플로우

결제 시작 후 실패·중단 시 동일 booking으로 재시도 허용 규칙.

#### 재시도 정책

| 항목 | 값 |
|---|---|
| 같은 booking 재시도 허용 창 | **30분** (최초 `REQUESTED` 생성 시점부터) |
| 최대 재시도 횟수 | **3회** |
| 3회 초과 or 30분 경과 시 | booking `EXPIRED` 전이 → 재예약 필수 |
| 슬롯 락 유지 | 30분 창 동안 슬롯 유지 (다른 멘티가 같은 시간에 예약 불가) |

#### 실패 유형별 처리

| 실패 유형 | 처리 |
|---|---|
| 사용자 중단 (뒤로가기/창닫기) | `payments.status = FAILED` 기록, booking은 `REQUESTED` 유지. 재시도 가능 |
| 카드 한도 초과 / 잔액 부족 | 에러 메시지 명확히 표시, 재시도 가능 |
| PG 시스템 장애 | 재시도 가능하되 5분 백오프 권장 UI |
| 이미 승인된 결제 중복 호출 | 멱등성 처리 (토스 `ALREADY_PROCESSED_PAYMENT`) — 기존 payment 레코드 참조 |
| 결제 승인 후 booking 업데이트 실패 | **보상 트랜잭션**: 자동 환불 + 관리자 알림 (데이터 정합성 우선) |

#### UX 요구사항

- 결제 실패 화면에 **"다시 시도"** 버튼 (30분 창 남아있을 때) / **"새 예약 만들기"** 버튼 (초과 시)
- 재시도 잔여 횟수 표시: "3회 중 1회 시도함"
- 30분 타이머 카운트다운 노출

#### 서버측 검증

- 재시도 시도 시 `booking.status === REQUESTED` 및 `created_at + 30min > now` 검증
- 재시도 횟수 카운트: `payments` 테이블에서 `booking_id` 기준 row 수
- 타임아웃 만료 booking은 스케줄러가 주기적으로 `EXPIRED` 전이 + 슬롯 해제

### 21-7. 원천징수 · 멘토 세무

플랫폼이 멘토에게 정산 시 원천징수 의무 발생. 정산 자동화는 Week 5+이지만 정책·계산 로직은 미리 확정.

#### 소득 분류 (세무사 자문 필수)

| 분류 | 원천징수율 | 적용 케이스 |
|---|---|---|
| **사업소득** | **3.3%** (소득세 3% + 지방세 0.3%) | 멘토 활동이 계속·반복적이고 인적용역 제공 (가장 일반적) |
| **기타소득** | **8.8%** (소득세 8% + 지방세 0.8%, 필요경비 60% 공제 후) | 일시적·우발적 소득. 연 300만원 이하 분리과세 가능 |

→ MVP는 **사업소득 3.3%**로 가정 (대부분의 플랫폼 표준). 런칭 전 세무사 자문으로 확정.

#### 정산 금액 계산 흐름

상품가 30,000원 기준 예시 (사업소득 3.3% 가정):

```
상품가              30,000원
└ 플랫폼 수수료    -15,000원 (50%)
멘토 총 보상       15,000원
└ 원천징수         -495원 (3.3%)
└ 패널티 차감      -0원 (해당 시 §20-5 적용)
멘토 실수령액     14,505원
```

DB 계산 (`bookings` + `mentor_warnings` 기반):
```
mentor_payout (실수령액) = (price - platform_fee) * (1 - withholding_rate) - penalty_deduction
withheld_tax = (price - platform_fee) * withholding_rate
```

#### 운영 의무

| 항목 | 내용 |
|---|---|
| **원천징수 신고** | 매월 다음달 10일까지 국세청에 일괄 신고·납부 (홈택스) |
| **지급명세서** | 매년 2월 말까지 전년도 멘토별 지급 내역 제출 |
| **연 300만원 초과** | 멘토 본인이 종합소득세 신고 의무 (5월). 약관 T07에 명시 |
| **신분증·계좌 사본** | 정산 시작 전 1회 확보. 본인인증과 별개 |
| **부가세** | 멘토 개인은 원칙적 면세, 플랫폼 수수료는 부가세 과세 (회사 사업자 기준) |

#### MVP 단계 단순화

- 자동 원천징수 시스템은 Week 5+ (정산 자동화에 묶음)
- MVP 기간엔 멘토 인원·금액 적어 **수동 정산** (엑셀 + 은행 이체) 가능
- 단 **약관(T07)에 원천징수 사실은 미리 명시**해서 멘토가 알고 있어야 함
